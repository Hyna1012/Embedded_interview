# 项目深挖优化问答总表

> 用途：把简历中的三个项目和美的实习统一成“面试官追问 - 你的回答”的形式。重点是回答要具体到链路、数据、模块和排查方法，让面试官感觉这是你自己做过、调过、复盘过的。

## 1. 总体自我介绍追问

### Q1. 你简历上项目比较多，怎么用 1 分钟讲清楚你的主线？

**参考回答：**  
我的主线是嵌入式软件和毫米波雷达应用落地。跌倒检测项目体现的是从雷达采集、信号处理到轻量化模型部署的端侧识别链路；生命体征项目体现的是 `ARC + FreeRTOS` 下的任务链路、多目标跟踪和嵌入式算法实现；多普勒项目体现的是更底层的 `AFE / ADC` 配置、`frame/chirp` 采样时序和原始帧输出。美的实习则让我接触到真实产品里的状态机、安全联锁、继电器控制和样机问题闭环。

### Q2. 你最想让面试官记住你哪三个关键词？

**参考回答：**

1. **板端链路完整**：做过采集、处理、模型部署、协议输出。
2. **底层调试意识**：能讲 `AFE / ADC`、采样时序、队列、帧封装、总线波形。
3. **工程落地经验**：美的实习接触过安全联锁、状态机、样机联调和缺陷闭环。

### Q3. 如果面试官问“你这些项目是不是都很像”，怎么区分？

**参考回答：**  
我会按技术重心区分。跌倒检测项目更偏端侧识别，重点是雷达信号处理、轻量化模型训练和 MCU 端部署；生命体征项目更偏实时任务链路和多人感知应用，重点是 `ARC + FreeRTOS`、点云跟踪、静止目标维护和呼吸心跳检测；多普勒项目更偏底层驱动和采集链路，重点是 `AFE / ADC`、`frame/chirp` 采样时序、`rawRadarCube` 和原始帧输出；美的实习则是产品电控，重点是状态机、安全联锁、继电器控制和缺陷闭环。它们不是同一套内容换名字，而是覆盖了嵌入式开发从底层采集、任务组织、算法落地到产品联调的不同侧面。

### Q4. 你的项目里最能体现“工程化”的共性是什么？

**参考回答：**  
我觉得共性是都不是只看最终结果，而是有中间状态和验证闭环。跌倒项目里通过 `TLV` 看原始采样、MTI、距离向量和分类概率；多普勒项目里看原始帧、`rawRadarCube`、距离维/速度维峰值；生命体征项目里看点云、目标 ID、静止目标状态、微动波形和频谱结果；美的实习里看输入、状态机、输出和样机响应。这样做的好处是出问题时能分层定位，而不是凭感觉改代码。

## 2. 跌倒检测项目深挖

### Q5. 你在跌倒检测项目里具体负责什么？

**参考回答：**  
我负责的是从板端数据到端侧推理的主链路。底层上，基于 `AT32F435 / RKB2301` 做板级初始化、雷达采集链路和数据通信；信号处理上，用 `CMSIS-DSP` 完成均值去除、窗函数、FFT、静态杂波抑制/MTI 和距离特征提取；算法上，对比 `CNN / GRU / LSTM`，完成参数导出和 MCU 端推理集成；联调上，通过 `TLV` 输出原始采样、中间特征和分类结果，用来验证板端和离线结果是否一致。

### Q6. 这个项目最能体现你嵌入式能力的点是什么？

**参考回答：**  
不是训练了一个模型，而是把模型真正放到 MCU 链路里跑起来。难点在于板端采样参数、信号处理流程、特征维度、滑窗长度和模型输入必须和训练侧一致。只要其中一个环节不一致，离线效果再好，板端结果也会漂。所以我会输出中间结果逐层对比，而不是只看最后分类。

### Q7. 跌倒项目如果识别效果不好，你怎么排查？

**参考回答：**  
我会分三层。第一层看采集链路：AFE、ADC、采样 buffer、帧同步是否正常；第二层看信号处理：均值去除、窗函数、FFT、MTI、距离向量是否和 Python 离线处理一致；第三层看模型部署：输入顺序、归一化、滑窗、参数导出、推理结果是否一致。这样可以避免一上来就怀疑模型。

### Q8. 为什么要输出中间结果？

**参考回答：**  
因为中间结果是定位问题的抓手。比如最终判断错了，我需要知道是原始数据异常、MTI 后目标变化不明显、距离向量维度错了，还是模型概率本身不稳定。`TLV` 输出原始采样、距离特征和分类概率，就是为了让每层都有可观测结果。

### Q9. 跌倒项目最容易被追问的“坑”是什么？

**参考回答：**  
最容易被追问的是“离线训练和板端推理怎么保证一致”。我会重点讲特征维度、数据顺序、归一化、滑窗长度和参数导出。因为模型能跑起来不代表结果可信，真正难的是让 C 侧输入和 Python 训练输入完全对齐。如果面试官继续追问，我会讲通过输出中间特征和概率对比来定位差异。

### Q10. 这个项目如果继续优化，你会从哪里下手？

**参考回答：**  
我会优先做三件事。第一，增加更多动作和边界场景数据，比如快速坐下、弯腰、躺下，减少误判；第二，优化模型输入特征和滑窗策略，让端侧推理更稳；第三，进一步压缩模型和中间 buffer，降低 RAM 占用和推理延迟。也就是说，不只是追求精度，而是从数据、模型和嵌入式资源三方面优化。

## 3. 生命体征项目深挖

### Q11. 生命体征项目你具体负责什么？

**参考回答：**  
我负责系统联调和嵌入式算法实现。具体到链路，是基于 `ARC + FreeRTOS` 做帧完成队列处理、用户任务组织和多模块数据流衔接；算法上围绕 `4D` 点云和扩展卡尔曼滤波做多人目标跟踪、静止目标维护和目标状态输出；生命体征部分针对胸腔微动信号做 `EDACM` 解调、一阶线性补偿和 `VMD` 频谱分离，输出呼吸心跳频率和波形。

### Q12. 为什么这个项目要先跟踪目标，再检测生命体征？

**参考回答：**  
因为多人场景下，必须先知道每个人在哪里，才能对对应目标提取呼吸和心跳。生命体征信号来自很小的胸腔位移，如果没有目标位置和 ID，系统不知道该从哪个距离单元或哪个目标区域提取相位。先跟踪再检测，可以做到分目标检测，而不是对所有距离单元盲目做频谱分析。

### Q13. 静止目标维护为什么重要？

**参考回答：**  
呼吸心跳检测通常在人相对静止时做，但人静止后点云可能减少，跟踪算法容易认为目标消失。静止目标维护就是为了保留目标状态和距离位置，让后续微动相位提取有稳定对象。否则系统可能刚要测生命体征，目标就被删掉。

### Q14. `EDACM + 线性补偿 + VMD` 这条链路怎么讲？

**参考回答：**  
先通过目标跟踪确定目标距离，再从对应位置提取相位微动信号；`EDACM` 用来得到更连续的相位微动，减少普通相位解算的跳变；一阶线性补偿用来去掉缓慢漂移；`VMD` 再把微动信号分成不同频率模态，从中提取呼吸和心跳频率。也就是说，这条链路不是堆算法名，而是围绕“从胸腔微动中稳定分离呼吸心跳”设计的。

### Q15. 生命体征结果不稳定怎么排查？

**参考回答：**  
先看目标跟踪是否稳定，包括目标 ID、位置和状态；再看静止目标维护有没有误删；然后看相位提取是否取到了正确距离位置；最后才看 `EDACM`、线性补偿和 `VMD` 参数。很多时候结果不稳不是频谱算法本身，而是前面的目标状态或输入信号不稳定。

### Q16. 生命体征项目最容易被质疑什么？怎么回答？

**参考回答：**  
容易被质疑的是“你是不是只是做了上位机显示或材料整理”。我会把回答拉回下位机任务链路：基于 `ARC + FreeRTOS` 做帧完成队列处理、用户任务组织和多模块数据流衔接；围绕 `4D` 点云和扩展卡尔曼滤波实现目标跟踪和静止目标维护；再对微动相位做 `EDACM`、线性补偿和 `VMD`。也就是说，我讲的是数据从帧完成事件进入用户任务，再到目标状态和生命体征结果输出的链路。

### Q17. 如果面试官问 FreeRTOS 在项目里的价值，你怎么答？

**参考回答：**  
FreeRTOS 的价值是把实时数据流组织清楚。底层处理完成一帧后，通过帧完成队列通知用户任务；用户任务再串接点云、目标跟踪、静止目标维护和生命体征计算。这样底层采集/处理不会和上层算法强耦合，调试时也能分清楚是队列事件、目标状态还是算法输出的问题。

## 4. 多普勒数据采集项目深挖

### Q18. 多普勒项目你现在应该怎么讲？

**参考回答：**  
我会把它讲成毫米波雷达底层驱动与多普勒数据采集系统，而不是以上位机或模拟器为主。我的重点是 `AFE / ADC` 参数配置、板级初始化、`frame/chirp` 采样时序控制、ADC 采样完成事件处理、`rawRadarCube` 数据组织和基于 GPIO 模拟 SPI 时序的数据输出。

### Q19. `rawRadarCube` 是什么？

**参考回答：**  
它是一帧雷达原始数据缓存。当前工程里每帧 `32` 个 chirp，每个 chirp `512` 点 ADC 采样，所以可以理解成 `32 × 512` 的二维数组。第一维是慢时间 chirp，第二维是快时间采样点。这个组织方式对应后续距离维 FFT 和多普勒维 FFT。

### Q20. 为什么不直接说 SPI，而说 GPIO 模拟 SPI 时序？

**参考回答：**  
因为当前工程里用的是 `SoftSPI`，本质是 GPIO 翻转模拟 SPI 时序，不是硬件 SPI 外设。直接写 SPI 容易被追问硬件 SPI 控制器和 DMA 配置。说“GPIO 模拟 SPI 时序”更准确，也体现我知道实现边界。

### Q21. 原始帧解析错位怎么排查？

**参考回答：**  
先保存原始字节流，检查 magic word 是否稳定出现；再核对总长度和实际接收长度；然后检查类型字段、长度字段、端序和结构体对齐；最后确认接收侧是否丢字节或多线程读写冲突。不要一开始就改热图算法。

### Q22. 多普勒项目如果被问“是不是只做了 Python”，怎么回答？

**参考回答：**  
不是。Python 只是验证侧，核心在下位机采集链路。下位机要完成 `AFE / ADC` 参数配置、`frame/chirp` 时序控制、ADC 采样完成事件处理、`rawRadarCube` 原始数据组织和帧输出。Python 侧用来解包和做二维 FFT，目的是验证下位机输出的数据是否能支撑距离-速度分析。

### Q23. 多普勒项目最能体现底层能力的地方是什么？

**参考回答：**  
是把采样时序、ADC buffer、原始帧组织和通信输出打通。这里不是简单调用一个外设接口，而是要保证每个 chirp 采样完成后数据写到正确位置，整帧结束后长度和帧格式正确，接收侧能恢复出完整 `32 × 512` 数据。这个链路稳定了，后续热图验证才有意义。

## 5. 美的实习深挖

### Q24. 美的实习你具体做了什么？

**参考回答：**  
我参与微波炉主控板嵌入式软件开发，基于 `NXP 56F8014` 做显示交互、输入检测、继电器控制和安全联锁等功能调试；负责状态机流程和异常保护逻辑实现，参与样机联调、功能测试、问题定位和软件缺陷闭环。

### Q25. 门联锁为什么不是普通 GPIO？

**参考回答：**  
因为它直接关系到安全。普通按键误判可能影响体验，但门联锁误判可能影响关键输出安全。软件上要保证门状态异常时关键输出优先断开，同时状态机进入正确的安全路径。面试时我会强调：门联锁是安全条件，不只是输入事件。

### Q26. 样机联调怎么体现工程能力？

**参考回答：**  
样机联调面对的是系统现象，不是单个函数。比如不启动可能是输入没识别、门联锁没满足、状态机没进入运行态、继电器输出没开，也可能是功率板执行侧没响应。排查时要按输入、状态、输出、硬件响应分层看。

### Q27. 软件缺陷闭环怎么讲？

**参考回答：**  
我理解缺陷闭环包括复现问题、确认触发条件、定位根因、修改代码、回归验证和版本确认。比如显示异常不一定是显示函数错，也可能是状态切换标志位没清；输入误判也可能影响状态机和输出控制。所以闭环不是改一行代码，而是确认问题在真实场景下被解决且没有引入新问题。

### Q28. 实习里如果被追问“你写了哪些代码”，怎么稳妥回答？

**参考回答：**  
我会围绕主控板控制逻辑说，不夸大成独立负责整机软件。具体包括显示交互、输入检测、状态机流程、异常保护和继电器控制相关功能调试。比如根据不同状态处理启动、暂停、开门、定时和故障场景；在样机联调中根据测试反馈定位显示、输入识别或联锁控制问题，再做修改和回归验证。

### Q29. 美的实习和学校项目相比，你最大的成长是什么？

**参考回答：**  
学校项目更关注方案能不能跑通，实习让我更关注功能在真实产品场景下是否稳定、安全、可回归。比如微波炉里开门、暂停、故障这些不是普通分支，而是必须优先保证安全态的状态；显示和输入也不是孤立模块，而是和状态机、联锁、继电器输出联动。这让我更理解嵌入式产品开发里的工程边界。
