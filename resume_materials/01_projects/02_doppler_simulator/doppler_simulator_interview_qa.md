# 毫米波雷达底层驱动与多普勒数据采集系统面试问答

> 当前口径：这个项目主要讲下位机底层驱动和采集链路，不主动讲“板端生成热图”“TDM-MIMO 虚拟通道补偿”“UART 输出热图”。Python 侧只作为验证工具出现。

## 1. 这个项目一句话怎么介绍？

这是一个基于 `24GHz FMCW` 雷达平台的底层驱动与多普勒数据采集项目，重点是把雷达 `AFE / ADC` 配置、`frame/chirp` 采样时序、原始数据缓存和帧输出链路打通，为后续距离-速度验证提供稳定原始数据。

## 2. 你的个人贡献是什么？

我主要参与下位机底层驱动和采集链路开发，工作包括雷达 `AFE / ADC` 参数配置、板级初始化、采样时序控制、ADC 采样完成事件处理和原始帧输出调试。项目里还结合 Python 验证侧完成原始帧解包和距离-速度结果校验，但主线还是下位机采集链路。

## 3. 为什么这个项目不重点写“模拟器”？

因为从当前简历定位看，我更想突出嵌入式软件能力。模拟器和热图验证能说明我理解多普勒原理，但更能体现岗位匹配度的是底层驱动、采样时序、数据缓存、帧封装和硬件通信调试这些内容。

## 4. 这个项目和跌倒检测项目怎么区分？

跌倒检测项目更偏端侧识别，重点是雷达信号处理、轻量化模型训练和 MCU 端推理部署；多普勒项目更偏底层采集链路，重点是让雷达原始数据稳定采回来、组织成帧并输出，为后续距离-速度验证提供数据源。两者都用到毫米波雷达，但一个偏应用识别，一个偏底层驱动和采集验证。

## 5. 下位机主流程是什么？

下位机从 `main.c` 进入后，先完成扩展 SRAM、BSP、GPIO 模拟 SPI 时序和帧封装相关初始化，然后启动传感器。主循环中持续调用 `Sensor_Process()`，根据消息处理单个 chirp 采样完成和整帧采集完成事件，最后把整帧原始数据封装输出。

## 6. 为什么要做扩展 SRAM？

因为项目需要缓存一整帧原始 ADC 数据。当前参数下每帧包含 `32` 个 chirp，每个 chirp `512` 点采样，组织成 `rawRadarCube[32][512]` 后会占用一定 RAM。扩展 SRAM 的意义是保证采集缓存和后续数据搬运有足够空间，避免内存不足影响采集稳定性。

## 7. `rawRadarCube` 是什么？

`rawRadarCube` 可以理解成一帧雷达原始采样数据的二维缓存。第一维对应 chirp 序号，第二维对应当前 chirp 内的 ADC 采样点。当前工程中它是 `rawRadarCube[CHIRPS][ADC_SAMPLES]`，也就是 `32 × 512` 的原始数据组织方式。

## 8. 单个 chirp 采样完成后做什么？

当收到单个 chirp 的 ADC 采样完成消息后，下位机会从 ADC buffer 中取出本次采样数据，并搬运到 `rawRadarCube` 对应的 chirp 行。这样一帧内多个 chirp 依次写入，就能形成后续距离-速度处理所需的慢时间数据序列。

## 9. 一帧采集完成后做什么？

当收到整帧采集完成消息后，下位机会封装帧头、数据长度、类型和长度字段，然后把 `rawRadarCube` 中的完整原始数据输出。输出完成后再停止当前帧流程，等待下一轮采集或状态控制。

## 10. `frame/chirp` 时序怎么理解？

`chirp` 是一次扫频采样过程，`frame` 是由多个 chirp 组成的一帧数据。距离信息主要来自单个 chirp 内的快时间采样，速度信息需要利用多个 chirp 之间的慢时间相位变化。因此下位机必须保证 chirp 调度、ADC 采样和帧结束事件之间的顺序稳定。

## 11. 这个项目里 `AFE / ADC` 配置包括什么？

`AFE` 配置主要与雷达前端工作状态、发射接收链路和采样流程相关；`ADC` 配置主要包括采样点数、采样 buffer、采样触发和完成事件。简历里不需要把寄存器细节全写出来，面试时可以重点讲“参数配置 - 采样触发 - buffer 获取 - 数据搬运”这条链路。

## 12. 为什么简历里建议写“GPIO 模拟 SPI 时序”，而不是直接写 `SPI`？

因为当前工程里的 `SoftSPI` 本质是用 GPIO 翻转模拟 `SPI` 时序，不是 MCU 硬件 SPI 外设。如果简历直接写 `SPI`，面试官可能追问硬件 SPI 控制器、DMA 或片选配置，容易把问题引到不准确的实现上。写“基于 GPIO 模拟 SPI 时序”更严谨。

## 13. `SoftSPI` 面试时怎么解释？

我会说它是软件模拟 SPI，也就是通过 GPIO 控制时钟线、数据线等时序，把数据按类似 SPI 的方式输出。它的优点是引脚和时序控制更灵活，适合当前验证链路；缺点是效率和 CPU 占用通常不如硬件 SPI，所以正式产品里要根据带宽和实时性再评估。

## 14. 为什么要设计帧封装格式？

原始 ADC 数据是连续二进制流，如果不加帧头、长度和类型字段，接收侧很难判断一帧从哪里开始、数据有多长、当前数据代表什么。帧封装可以提升同步、解析和调试效率，也方便后续扩展不同类型的数据。

## 15. 帧格式里最关键的字段是什么？

最关键的是 magic word、总长度、类型和长度。magic word 用于接收侧找帧头，总长度用于判断整帧是否收完整，类型用于区分数据内容，长度用于跳过或解析当前数据段。这样接收侧不需要靠猜测固定位置解析。

## 16. 原始数据为什么不直接在板端处理成热图？

当前项目重点是底层采集链路验证，所以优先保证原始数据采集、缓存和输出稳定。把二维 FFT 和热图显示放在 Python 侧，更方便快速验证参数、观察结果和定位问题。等采集链路稳定后，再考虑是否把更多处理放到板端。

## 17. Python 侧在这个项目里做什么？

Python 侧主要是验证工具，不是项目主角。它负责解析下位机输出的原始帧，对数据做均值去除、距离维 FFT、多普勒维 FFT和静态杂波抑制，生成 `Range-Doppler Map`，用来验证目标距离、速度与热图峰值位置是否对应。

## 18. 为什么还要做多普勒仿真？

仿真能提供一个可控参考。通过仿真可以先建立目标距离、目标速度、`IF` 频率和多普勒频率之间的理论关系，再和真实采集数据处理后的结果对照。如果结果不一致，就能判断问题更可能出在参数配置、采样链路、帧解析还是理论理解上。

## 19. 如果采到的数据全是异常值，你会怎么排查？

我会先从底层开始排查：第一看雷达前端和 ADC 参数是否配置正确；第二看采样触发和完成事件是否按预期产生；第三看 ADC buffer 地址和搬运长度是否正确；第四看帧封装长度和接收侧解析是否一致；最后再看验证侧处理参数是否匹配。

## 20. 如果接收侧解析总是错位，怎么定位？

先检查 magic word 是否稳定出现，再核对帧总长度和实际接收长度是否一致；然后检查类型和长度字段是否按小端格式解析；最后用保存的原始字节流离线解析，排除实时显示层或串口/采集线程带来的干扰。

## 21. 如果热图峰值位置不对，怎么排查？

先确认下位机采样参数是否和 Python 处理参数一致，包括采样点数、chirp 数、FFT 点数、扫频参数和帧周期；再检查原始帧是否丢数据或错位；最后结合仿真结果判断距离维或速度维映射是否有参数换算错误。

## 22. 这个项目中最能体现底层驱动能力的点是什么？

我认为是把 `AFE / ADC` 配置、采样事件、数据缓存和 GPIO 模拟 SPI 时序输出串成稳定链路。这里不只是写一个接口函数，而是要保证采样时序、内存搬运、帧边界和数据输出都能对上。

## 23. 面试官问“你这个项目是不是只是上位机画图”，怎么回答？

不是。上位机画图只是验证手段，项目核心在下位机侧。真正要解决的问题是雷达是否能按设定时序采样，ADC 数据能否稳定进入 `rawRadarCube`，整帧原始数据能否按协议正确输出。没有下位机这条链路，上层热图也没有可靠数据源。

## 24. 这个项目目前不建议主动讲哪些内容？

不建议主动讲“板端生成 `Range-Doppler Map`”“`2Tx3Rx TDM-MIMO` 虚拟通道补偿”“`UART5` 输出热图”等内容。除非后续确认对应分支或实物代码，否则简历和面试都应该围绕底层驱动、原始数据采集和距离-速度验证展开。

## 25. 这个项目最适合提炼成哪些关键词？

`24GHz FMCW`、`AT32F435 / RKB2301`、`AFE / ADC`、板级初始化、`frame/chirp` 采样时序、ADC 采样完成事件、`rawRadarCube`、原始帧封装、GPIO 模拟 SPI 时序、距离-速度映射验证。

## 26. 一句话卖点怎么说？

这是一个围绕 `24GHz FMCW` 雷达的底层驱动与原始数据采集项目，我主要参与下位机采样时序、ADC 数据组织和原始帧输出链路，为后续距离-速度验证提供稳定数据源。

## 27. 面试前复习顺序

1. `main.c`：启动流程和主循环
2. `bsp_config.c`：采样参数、帧参数、ADC buffer
3. `bsp.c`：`Sensor_Process`、chirp/frame 消息处理、`rawRadarCube`
4. `tlv.c`：帧头、类型长度字段、原始数据输出
5. `drv_soft_spi.c`：GPIO 模拟 SPI 时序
6. `frameData.py`：Python 原始帧解包和二维 FFT
7. `simuDoppler.py / simuRangeDoppler.py`：距离-速度映射仿真
