# MCU 启动、异常与基础外设问答

> 用途：专门补 MCU 上电启动、向量表、异常、时钟树、看门狗、定时器、GPIO 等基础题。

## 1. 使用原则

- 这类题的重点不是背名词，而是能解释“程序为什么能从上电跑到 `main`”。
- 回答异常题时，要体现定位思路：看现场、查地址、缩小边界。
- 外设题尽量结合按键、门联锁、蜂鸣器、显示刷新、运行计时等真实场景。

## 2. 高频问答

### Q1. MCU 上电后大概发生什么？

**直接回答：**
上电后先取向量表里的初始栈指针和复位入口，然后执行启动文件，最后进入 `main`。

**展开解释：**
启动流程通常包括设置栈指针、复制 `data` 段到 RAM、清零 `bss` 段、初始化系统时钟和 C 运行环境，再调用 `main`。

**工程补充：**
如果程序跑不到 `main`，排查方向通常是启动文件、链接脚本、向量表、时钟、供电复位和下载地址。

### Q2. 向量表是什么？

**直接回答：**
向量表是一张保存异常和中断入口地址的表。

**展开解释：**
Cortex-M 向量表开头通常是初始 `MSP`，后面是 `Reset_Handler`、`NMI`、`HardFault` 以及外设中断入口地址。

**工程补充：**
Bootloader 跳转 App 时要特别注意重定位向量表，否则中断可能还跳到 Bootloader 的处理函数。

### Q3. `HardFault` 常见原因有哪些？

**直接回答：**
常见原因包括非法地址访问、栈溢出、函数指针错误、未对齐访问、执行非法指令。

**展开解释：**
比如访问空指针或野指针、数组越界破坏返回地址、任务栈太小、跳转到错误地址，都可能触发 `HardFault`。

**工程补充：**
排查时先看故障栈帧里的 `PC/LR`，再结合 `map` 文件和反汇编定位出错位置；别只盯着最后停住的异常函数。

### Q4. 什么是时钟树？为什么重要？

**直接回答：**
时钟树描述 MCU 各模块时钟来源、倍频分频和分发关系。

**展开解释：**
CPU、总线、定时器、串口、ADC 等外设都依赖时钟。时钟配置不对，外设波特率、采样率、PWM 频率都会偏。

**工程补充：**
调串口乱码、ADC 采样率异常、定时器周期不对时，除了看外设寄存器，也要先确认系统时钟和外设时钟。

### Q5. 看门狗的作用是什么？

**直接回答：**
看门狗用于在程序异常卡死时自动复位系统。

**展开解释：**
程序需要周期性喂狗，如果长时间没有喂狗，说明主流程可能跑飞或阻塞，看门狗会触发复位。

**工程补充：**
喂狗点不能随便放在中断或定时器里，否则主任务死了也可能继续喂狗；更稳的是让关键任务都正常运行后再统一喂狗。

### Q6. 定时器可以做哪些事情？

**直接回答：**
定时器可以做周期中断、PWM 输出、输入捕获、输出比较和时间测量。

**展开解释：**
周期中断适合调度低频任务；PWM 适合电机、蜂鸣器、背光等控制；输入捕获可以测频率、脉宽；输出比较可以做精确事件触发。

**工程补充：**
美的电控场景里，定时器常和按键消抖、蜂鸣器、显示刷新、运行计时和控制时序相关。

### Q7. PWM 的核心参数是什么？

**直接回答：**
PWM 核心参数是频率和占空比。

**展开解释：**
频率决定周期长短，占空比决定高电平时间比例。通过调占空比，可以改变等效输出能量或控制量。

**工程补充：**
面试里可以结合蜂鸣器、背光、电机调速或功率控制去讲，不要只背公式。

### Q8. GPIO 输入为什么要考虑上拉下拉？

**直接回答：**
因为悬空输入会受到干扰，电平不稳定。

**展开解释：**
上拉/下拉给输入引脚一个默认电平，外部信号改变时再切换状态。没有默认电平时，按键、门开关等输入可能抖动或误判。

**工程补充：**
门联锁、按键输入这类安全或交互信号，除了上拉下拉，还要考虑消抖、滤波和异常默认安全态。

### Q9. `data` 段为什么要从 Flash 拷贝到 RAM？

**直接回答：**
因为已初始化的全局变量运行时要放在 RAM，但它们的初始值需要存放在 Flash 中。

**展开解释：**
Flash 掉电不丢，适合保存初始值；RAM 可读写，适合运行时变量。启动文件会把 Flash 中保存的初始值复制到 RAM 对应的 `.data` 区域。

**工程补充：**
如果启动文件或链接脚本配置错误，可能导致全局变量初值不对，程序一开始就异常。

### Q10. `bss` 段为什么要清零？

**直接回答：**
因为未初始化或初始化为 0 的全局/静态变量，C 语言要求启动后默认为 0。

**展开解释：**
`.bss` 不需要在 Flash 里存一堆 0，启动时统一把 RAM 中对应区域清零即可。

**工程补充：**
这也是为什么启动文件不只是跳到 `main`，它还要完成 C 运行环境准备。

### Q11. `MSP` 和 `PSP` 分别是什么？

**直接回答：**
`MSP` 是主栈指针，`PSP` 是进程栈指针。

**展开解释：**
Cortex-M 上异常和中断默认使用 `MSP`；RTOS 里每个线程通常使用自己的 `PSP`，这样任务栈彼此独立。

**工程补充：**
理解 `MSP/PSP` 有助于理解 RTOS 任务切换和异常现场保存。

### Q12. `Reset_Handler` 是什么？

**直接回答：**
`Reset_Handler` 是 MCU 复位后执行的入口函数。

**展开解释：**
CPU 从向量表中取出复位入口地址，跳到 `Reset_Handler`。它通常负责初始化数据段、清零 bss、调用系统初始化，再进入 `main`。

**工程补充：**
如果换 Bootloader、改 App 地址或修改链接脚本，要确认向量表里的复位入口是否正确。

### Q13. 为什么会栈溢出？

**直接回答：**
栈溢出通常是因为局部变量太大、函数调用太深、递归或任务栈分配太小。

**展开解释：**
栈用于保存局部变量、返回地址、寄存器现场等。如果栈空间不够，就会覆盖其他内存，导致随机异常。

**工程补充：**
RTOS 里每个任务都有自己的栈，要结合栈水位查看是否够用；大数组尽量不要放在任务局部变量里。

### Q14. `HardFault` 发生后你怎么定位？

**直接回答：**
先保留故障现场，再根据栈帧里的 `PC/LR` 找到出错代码位置。

**展开解释：**
`HardFault` 发生时，硬件会压栈部分寄存器，包括 `R0-R3`、`R12`、`LR`、`PC`、`xPSR`。其中 `PC` 通常指向异常发生位置附近。

**工程补充：**
定位时可以结合 `map` 文件、反汇编和调用栈。如果 `PC` 是非法地址，要怀疑函数指针、栈破坏或跳转地址错误。

### Q15. 为什么有时候程序跑飞后还能被看门狗拉回来？

**直接回答：**
因为看门狗独立计时，程序异常后无法按时喂狗，就会触发复位。

**展开解释：**
看门狗是对系统活性的兜底机制，不能防止 Bug，但能避免系统永久卡死。

**工程补充：**
更好的设计是复位后记录复位原因和关键状态，便于后续分析，而不是只让系统“看起来恢复了”。

### Q16. MCU 启动慢可能有哪些原因？

**直接回答：**
可能是时钟稳定等待、外设初始化耗时、自检流程、Flash 读写、通信等待或启动日志过多。

**展开解释：**
有些系统上电要等待外部晶振、PLL 锁定、传感器复位、屏幕初始化或参数校验，这些都会影响启动时间。

**工程补充：**
产品里要区分必须阻塞等待的初始化和可以延后异步完成的初始化，避免不必要拖慢启动。

### Q17. 如果程序下载后不运行，但 Debug 能运行，可能是什么原因？

**直接回答：**
可能是启动配置、复位时序、时钟初始化、看门狗、未初始化变量或 Debug 环境改变了运行条件。

**展开解释：**
Debug 模式可能会暂停看门狗、初始化部分调试资源，或者改变时序。脱离调试器后，真实上电路径暴露出问题。

**工程补充：**
这类问题要按真实上电流程排查，不能只依赖单步调试。

### Q18. 什么是低电压复位？

**直接回答：**
低电压复位是在供电低于安全阈值时自动复位 MCU。

**展开解释：**
电压过低时，CPU、Flash、RAM 或外设可能不能可靠工作，低电压复位可以避免系统在不可靠电压下乱跑。

**工程补充：**
现场设备偶发重启时，电源跌落、负载启动冲击、继电器动作干扰都可能触发低电压复位。
