# 嵌入式 Linux 入门高频问答

> 用途：专门补嵌入式 Linux 基础概念，适合你简历里“了解 Linux 框架”这一层级。

## 1. 使用原则

- 当前建议把 Linux 表述为“了解框架”，不要说成主力项目经验。
- 面试中如果被追问 Linux，先从用户态/内核态、设备树、字符设备、驱动接口这些基础概念回答。
- 不确定的内核细节不要硬编，重点体现后续学习路径清晰。

## 2. 高频问答

### Q1. 嵌入式 Linux 和裸机/RTOS 最大区别是什么？

**直接回答：**
Linux 有完整内核、进程隔离和文件系统，裸机/RTOS 更轻量、更贴近硬件实时控制。

**展开解释：**
Linux 适合复杂应用、网络、多进程和文件系统场景；裸机/RTOS 适合资源受限、实时控制强、启动快的 MCU 场景。

**工程补充：**
你简历里对 Linux 更适合说“了解框架”，回答时可以从应用层、驱动层、设备树这些概念讲起。

### Q2. 用户态和内核态有什么区别？

**直接回答：**
用户态运行普通应用，内核态运行操作系统内核和驱动，权限更高。

**展开解释：**
用户程序不能直接随意访问硬件，需要通过系统调用进入内核，由驱动完成硬件操作。

**工程补充：**
这也是为什么 Linux 驱动要提供 `open/read/write/ioctl` 等接口，让应用通过文件抽象访问设备。

### Q3. 设备树是做什么的？

**直接回答：**
设备树用来描述硬件资源，让内核知道板子上有哪些设备以及它们怎么连接。

**展开解释：**
它通常描述寄存器地址、中断号、时钟、GPIO、总线关系等信息。驱动通过匹配设备树节点获得硬件配置。

**工程补充：**
设备树的价值是把“硬件描述”和“驱动代码”分开，同一驱动可以适配不同板级配置。

### Q4. 字符设备驱动是什么？

**直接回答：**
字符设备驱动是以字节流方式访问设备的 Linux 驱动类型。

**展开解释：**
应用层通常通过设备文件调用 `open/read/write/ioctl` 等接口，内核驱动再把这些操作转成硬件访问。

**工程补充：**
串口、按键、LED、简单传感器都可以用字符设备思路理解。

### Q5. `ioctl` 有什么用？

**直接回答：**
`ioctl` 用于传递不适合用普通 `read/write` 表达的控制命令。

**展开解释：**
比如设置工作模式、读取状态、配置参数、触发特殊动作等，都可以通过 `ioctl` 做控制。

**工程补充：**
普通数据流用 `read/write`，控制类命令用 `ioctl` 更合适。

### Q6. Linux 驱动和裸机驱动有什么区别？

**直接回答：**
裸机驱动通常直接操作硬件，Linux 驱动运行在内核框架下，要通过内核 API 和设备模型管理硬件。

**展开解释：**
Linux 驱动不仅要访问寄存器，还要处理设备树匹配、资源申请、中断注册、字符设备接口、同步互斥和用户态交互。

**工程补充：**
如果面试问 Linux，你可以稳妥说：“我目前 MCU 裸机/RTOS 更熟，Linux 驱动了解整体框架，知道应用层通过设备文件和系统调用访问驱动。”

### Q7. Linux 里“一切皆文件”怎么理解？

**直接回答：**
很多设备和资源都可以抽象成文件，通过统一的文件接口访问。

**展开解释：**
应用层可以用 `open/read/write/ioctl/close` 操作设备节点，内核驱动把这些文件操作映射到具体硬件行为。

**工程补充：**
这让应用不需要直接知道底层寄存器，只需要调用统一接口。

### Q8. `procfs` 和 `sysfs` 有什么用？

**直接回答：**
它们都是内核向用户态暴露信息的虚拟文件系统。

**展开解释：**
`procfs` 常用于进程和内核运行信息；`sysfs` 常用于设备、驱动、总线、类等内核对象的信息和属性。

**工程补充：**
驱动调试时，可以通过这些接口查看设备状态或暴露简单配置项。

### Q9. Linux 中断处理为什么常分上半部和下半部？

**直接回答：**
为了让中断快速响应，同时把耗时处理延后执行。

**展开解释：**
上半部处理中断里最紧急的部分，比如清中断、读取状态；下半部处理耗时逻辑，比如数据解析、通知用户态。

**工程补充：**
这和 MCU 里“ISR 里做轻活，线程里做重处理”的思路是一致的。

### Q10. `poll/select/epoll` 是做什么的？

**直接回答：**
它们用于让应用等待一个或多个文件描述符的事件。

**展开解释：**
比如等待串口、socket 或设备节点是否可读可写。`select` 和 `poll` 适合较少描述符，`epoll` 更适合大量连接场景。

**工程补充：**
嵌入式应用层如果要同时处理多个设备或网络连接，就会用到这类 I/O 多路复用机制。

### Q11. 阻塞 I/O 和非阻塞 I/O 有什么区别？

**直接回答：**
阻塞 I/O 在数据未就绪时会等待，非阻塞 I/O 会立即返回。

**展开解释：**
阻塞模型简单，但可能卡住当前线程；非阻塞模型需要应用自己处理“暂时没有数据”的返回值，通常配合 `poll/select/epoll`。

**工程补充：**
这和 MCU 里轮询、事件通知的取舍类似，本质都是在处理“等数据时 CPU 做什么”。

### Q12. 什么是交叉编译？

**直接回答：**
交叉编译是在一种平台上编译运行在另一种平台上的程序。

**展开解释：**
比如在 x86 PC 上编译 ARM Linux 板子上运行的程序，需要使用对应的交叉编译工具链。

**工程补充：**
嵌入式 Linux 开发常见流程是 PC 编译，目标板运行，所以工具链、库版本和目标架构要匹配。

### Q13. U-Boot、Kernel、RootFS 分别是什么？

**直接回答：**
U-Boot 是引导程序，Kernel 是 Linux 内核，RootFS 是根文件系统。

**展开解释：**
U-Boot 负责初始化硬件并加载内核；Kernel 负责进程、内存、驱动和文件系统管理；RootFS 提供用户态程序、库和配置文件。

**工程补充：**
这三者构成嵌入式 Linux 系统启动链路：Bootloader 引导内核，内核挂载根文件系统，再启动用户态应用。
