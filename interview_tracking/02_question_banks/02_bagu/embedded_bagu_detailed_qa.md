# 嵌入式八股详细回答

> 这份文件只保留通用八股内容，方便你单独背定义、机制和标准回答。
> 回答结构默认统一为：
> 1. 先直接回答
> 2. 再展开解释
> 3. 最后补工程补充

## 1. RTOS / RT-Thread

### Q1. 为什么项目里需要 RTOS，而不是裸机？

**直接回答：**
因为项目里存在实时性不同、职责不同的多个任务，直接裸机会让逻辑耦合很重。

**展开解释：**
像采集、处理、输出、联调这些任务关注点不一样，实时性要求也不一样。如果全部塞进一个主循环，后面一旦功能变多，逻辑会越来越乱，也不利于保障关键任务优先执行。引入 RTOS 后，可以用线程和优先级把它们拆开管理。

**工程补充：**
不要把 RTOS 讲成“高级框架”，要讲成“让任务组织更清晰、更可维护的工具”。

### Q2. 为什么选择 `RT-Thread`，而不是 `FreeRTOS`？

**直接回答：**
对我的项目来说，选择 `RT-Thread` 的核心原因是更方便做多任务组织、同步通信和系统工程化管理。

**展开解释：**
项目里采集、处理、输出和联调任务实时性不同，我需要用线程、优先级和同步机制把它们拆开。`RT-Thread` 更符合我当时的开发习惯和任务组织方式。`FreeRTOS` 当然也可以完成这些功能，所以这里不是说谁绝对更强，而是项目背景和使用习惯下的选择。

**工程补充：**
这题最稳的口径是：
“我选 RT-Thread 的核心理由是任务组织，不是为了某个花哨组件。”

### Q3. RT-Thread 调度器的核心机制是什么？

**直接回答：**
RT-Thread 调度器的核心是从当前可运行线程里选择优先级最高的线程执行。

**展开解释：**
线程状态变化后，系统会维护就绪线程集合；当需要调度时，选择当前最高优先级的就绪线程运行。如果有更高优先级线程变成就绪态，就会触发切换。

**工程补充：**
先讲调度本质，再讲 `PendSV` 是 Cortex-M 上的具体实现，不要一上来只背名词。

## 2. 中断 / DMA / Cache / MPU

### Q4. DMA 的根本价值是什么？

**直接回答：**
DMA 的根本价值不是单纯搬得更快，而是搬数据时不占 CPU。

**展开解释：**
DMA 负责把数据从外设搬到内存、或者从内存搬到外设，CPU 不用亲自参与每个数据搬运动作。这样 CPU 就可以把更多时间留给控制逻辑、处理任务和其他更有价值的计算工作。

**工程补充：**
一句话记忆：
“DMA 的核心价值是解放 CPU。”

### Q5. 为什么很多串口接收方案会用“DMA + 空闲中断”？

**直接回答：**
因为这种方案同时兼顾了低 CPU 占用和不定长收包的边界判断。

**展开解释：**
DMA 负责把串口数据直接搬到内存，减少每字节中断带来的 CPU 开销；空闲中断则在一帧数据结束后提供一个自然的“包尾”信号，让软件知道当前可以开始处理这一包数据。

**工程补充：**
可以把它概括成：
“DMA 负责搬运，空闲中断负责判包边界。”

### Q6. DMA 搬完数据后，CPU 为什么可能还读到旧数据？

**直接回答：**
因为 DMA 更新的是内存，而 CPU 可能还在读 Cache 里的旧副本。

**展开解释：**
这其实是典型的数据一致性问题。DMA 看见的是 RAM，CPU 读的时候可能优先命中 D-Cache，如果 Cache 没失效，就可能读到旧值。所以在带 Cache 的 MCU 上，DMA 和 CPU 并发访问同一块缓冲区时，必须考虑一致性维护。

**工程补充：**
标准说法就是：
“这是 DMA 和 Cache 之间的一致性问题，不是 DMA 本身没工作。”

### Q7. `Clean` 和 `Invalidate` 分别什么时候用？

**直接回答：**
`Clean` 一般用于让内存看到 CPU 刚写过的最新数据，`Invalidate` 一般用于让 CPU 丢掉旧 Cache、重新读内存。

**展开解释：**
如果是 CPU 先写，之后要让 DMA 或外设看到最新内存内容，就更偏向 `Clean`；如果是 DMA 先写完数据，之后要让 CPU 读取最新结果，就更偏向 `Invalidate`。核心是看“谁先写、谁后读”。

**工程补充：**
这题一定别只背定义，要配合 DMA 场景去讲。

### Q8. `MPU` 是什么？

**直接回答：**
MPU 是内存保护单元，用来把内存划分成不同区域，并定义不同访问属性。

**展开解释：**
它可以限制哪些区域可读写、可执行、可缓存、是否有特殊访问顺序要求。对很多 MCU 来说，MPU 不仅能做保护，也和 Cache、DMA 一致性配置有关。

**工程补充：**
一句话记忆：
“MPU 不只是保护内存，还帮助定义内存怎么被访问。”

## 3. 驱动 / 接口

### Q9. 如果给你一块新板子，你怎么 bring-up？

**直接回答：**
我会按“看资料 - 拉最小链路 - 做连通性验证 - 再补功能”的顺序来做。

**展开解释：**
第一步看原理图和数据手册，确认供电、时钟、复位、引脚和接口方式。第二步做最小化初始化，比如 GPIO、时钟和通信总线。第三步先做最简单的连通性测试，比如读 ID、读默认寄存器。只有最小链路跑通了，才继续写完整驱动和功能代码。

**工程补充：**
这题考的是工程思维，所以不要答成“先写代码”。

### Q10. `SPI` 和 `IIC` 的区别是什么？

**直接回答：**
`SPI` 更偏高速、点对点；`IIC` 更偏低引脚占用、多设备挂载。

**展开解释：**
`SPI` 速率高、协议简单、时序直接，适合高速配置和数据传输，但线多、片选管理复杂。`IIC` 两根线就能挂多个设备，资源占用低，但速率通常不如 `SPI`，而且协议里有起始、停止、`ACK/NACK` 等时序细节。

**工程补充：**
实际选型看速率要求、设备数量和板上资源。

### Q11. `UART` 波特率 `115200` 发 1 字节为什么不能只按 8 位算？

**直接回答：**
因为 UART 一帧通常不止 8 个数据位，还包含起始位、停止位，可能还有校验位。

**展开解释：**
也就是说，一个“字节”在串口线上传的时候，实际传输位数可能是 10 位、11 位甚至更多。所以估算时间时不能直接用 `1/115200`，而要按完整帧位数算。

**工程补充：**
一句口语化表达：
“串口发的是一帧，不只是 8 位数据本体。”

## 4. C 语言

### Q12. `sizeof` 和 `strlen` 的区别是什么？

**直接回答：**
`sizeof` 是编译期求对象占用字节数，`strlen` 是运行期求字符串长度。

**展开解释：**
`sizeof` 面向的是类型或对象本身，不关心里面是不是字符串；`strlen` 只适用于以 `\\0` 结尾的字符串，而且它统计的是字符个数，不包括结尾的 `\\0`。

**工程补充：**
这是特别高频的 C 题，最好背一个数组退化成指针的例子配合说。

### Q13. `volatile` 的作用是什么？

**直接回答：**
`volatile` 的作用是告诉编译器，这个变量可能被外部异步修改，不要做过度优化。

**展开解释：**
典型场景包括硬件寄存器访问、中断和主循环共享变量、多线程里被异步修改的状态变量。它的核心不是“线程安全”，而是“禁止编译器把访问优化得太激进”。

**工程补充：**
一定要记一句：
“`volatile` 解决的是可见性问题，不是同步互斥问题。”

### Q14. `malloc` 为什么可能带来内存碎片？

**直接回答：**
因为动态分配和释放反复发生后，空闲内存会被切成很多不连续的小块。

**展开解释：**
如果程序不断申请不同大小的内存，再以不规则顺序释放，长期运行后就可能出现“总空闲内存还不少，但没有一块足够大的连续区域可用”的情况。这就是典型碎片问题。

**工程补充：**
所以很多嵌入式项目会尽量少用动态内存，或者只在初始化阶段使用。

## 5. 网络

### Q15. `TCP` 和 `UDP` 的区别是什么？

**直接回答：**
TCP 是面向连接、可靠传输，UDP 是无连接、开销更小。

**展开解释：**
TCP 通过三次握手建立连接，保证数据有序、可靠到达，适合可靠性要求高的场景；UDP 不建立连接，不保证重传和顺序，但延迟和开销通常更小，更适合实时性要求高、允许少量丢包的场景。

**工程补充：**
最后一定补一句：
“具体选哪个，要看是更看重可靠性，还是更看重实时性和开销。”

### Q16. 为什么说 TCP 更可靠，但不一定更适合所有场景？

**直接回答：**
因为 TCP 的可靠性是有代价的，这个代价就是更高的时延、状态维护和协议开销。

**展开解释：**
TCP 通过连接管理、确认重传、顺序控制等机制保证可靠传输，这非常适合对完整性要求高的场景。但如果场景本身更看重实时性、容忍少量丢包，那么 UDP 反而可能更合适。

**工程补充：**
这题一定要体现“看场景选协议”的思维。
