# 大厂嵌入式软件面试真题方向整理

> 用途：根据常见大厂/头部硬件公司嵌入式面经主题整理，不绑定某一家公司原题，避免背偏题；重点训练“C 语言 + RTOS + MCU 外设 + 调试 + 项目深挖”的组合拳。

## 1. 高频考察画像

嵌入式软件岗大厂面试通常不会只问单点知识，常见组合是：

- C 语言底层：指针、数组、内存、`volatile`、`const`、结构体对齐、函数指针。
- MCU 基础：启动流程、中断、异常、NVIC、时钟、Flash、SRAM、栈。
- RTOS：任务状态、调度、优先级、队列、信号量、互斥锁、ISR 同步。
- 外设通信：UART、IIC、SPI、CAN、DMA、定时器、ADC。
- 调试能力：HardFault、死机、丢包、IIC 无 ACK、SPI 数据错、串口乱码。
- 项目深挖：你做了什么、为什么这么做、遇到什么问题、怎么定位、有什么取舍。
- 工程意识：实时性、可靠性、异常保护、资源约束、可维护性。

## 2. C 语言真题方向

### Q1. `volatile` 有什么用？项目里什么时候用？

**答法：**  
`volatile` 告诉编译器变量可能被当前代码流之外的因素修改，不要把它优化成寄存器缓存。典型场景是中断和主循环共享变量、硬件寄存器映射、多任务共享状态标志。  
项目里比如 ADC 采样完成标志、串口接收状态、门联锁输入状态，如果可能被中断或硬件异步改变，就要考虑 `volatile`。但 `volatile` 不等于线程安全，它不能替代锁、临界区或原子操作。

### Q2. 数组名和指针有什么区别？

**答法：**  
数组名在大多数表达式里会退化成指向首元素的指针，但数组本身不是指针。区别体现在 `sizeof`、可修改性和内存拥有关系上。比如 `sizeof(arr)` 得到整个数组大小，`sizeof(p)` 得到指针大小；数组名不能像普通指针变量一样重新赋值。

### Q3. 结构体对齐为什么重要？

**答法：**  
结构体对齐会影响内存占用、访问效率和协议解析。嵌入式里如果结构体直接映射硬件寄存器或通信协议帧，就必须关注字段大小、对齐和端序，否则可能出现解析错位。面试里可以结合 `TLV` 帧头说：协议帧最好明确每个字段宽度，不要随便依赖编译器默认结构体布局。

### Q4. `malloc` 为什么在嵌入式里要慎用？

**答法：**  
因为动态内存会带来碎片、分配失败和耗时不可控。长期运行的 MCU 项目里，内存碎片可能导致运行一段时间后才出问题，排查很困难。雷达项目中大 buffer、FFT buffer、原始帧缓存通常更适合静态分配。

### Q5. 栈和堆有什么区别？任务栈又是什么？

**答法：**  
栈用于函数调用、局部变量和上下文保存，生命周期自动管理；堆用于动态分配，生命周期由程序控制。RTOS 中每个任务都有自己的任务栈，所以任务里局部大数组会占用该任务栈，可能导致单个任务栈溢出，而不是整个系统 RAM 立刻不够。

## 3. RTOS 真题方向

### Q6. FreeRTOS 任务状态有哪些？

**答法：**  
常见状态有 `Running`、`Ready`、`Blocked`、`Suspended`。`Running` 是正在执行，`Ready` 是可运行但暂未获得 CPU，`Blocked` 是等待时间或事件，`Suspended` 是被显式挂起且不参与调度。

### Q7. 队列、信号量、互斥锁怎么选？

**答法：**  
要传数据用队列；只通知事件用二值信号量或任务通知；保护共享资源用互斥锁。互斥锁相比普通信号量更适合资源保护，因为它有所有权语义和优先级继承机制。

### Q8. 中断里能不能调用 FreeRTOS API？

**答法：**  
可以，但要调用 `FromISR` 版本，而且中断优先级要满足 FreeRTOS 对系统调用中断优先级的限制。中断里不应该做复杂业务逻辑，通常是清中断、发队列/信号量/任务通知，然后让任务处理。

### Q9. 优先级反转是什么？

**答法：**  
低优先级任务持有锁，高优先级任务等待锁，中优先级任务又抢占低优先级任务，导致高优先级任务被间接阻塞。解决思路是互斥锁优先级继承、缩短临界区、减少共享资源竞争。

### Q10. 项目里为什么要用队列？

**答法：**  
队列可以把生产者和消费者解耦。比如生命体征项目里，底层帧处理完成后通过帧完成队列通知用户任务，用户任务再处理点云、跟踪、目标维护和生命体征计算。这样不会让底层采集和上层算法强耦合。

## 4. MCU / 中断 / 启动真题方向

### Q11. MCU 上电后到 `main()` 前发生了什么？

**答法：**  
通常先从向量表取初始栈指针和复位入口，进入 Reset_Handler；然后设置时钟、初始化数据段 `.data`，清零 `.bss`，可能初始化 C 库，最后跳转到 `main()`。如果使用 RTOS，通常在 `main()` 里创建任务，最后启动调度器。

### Q12. 中断和异常有什么区别？

**答法：**  
中断通常来自外设或外部事件，比如 UART 接收、ADC 完成；异常更多来自内核或执行过程，比如 HardFault、SysTick、PendSV、SVC。二者都会打断当前执行流，但触发来源和语义不同。

### Q13. Cortex-M 里 `PendSV` 为什么适合做任务切换？

**答法：**  
因为 `PendSV` 可以被挂起，并通常配置为较低优先级。RTOS 可以先在 SysTick 或中断里标记需要切换，再让 `PendSV` 在合适时机统一保存/恢复上下文，避免在高优先级中断里做复杂切换。

### Q14. HardFault 怎么排查？

**答法：**  
先看 fault 寄存器和堆栈现场，确认是非法地址、总线错误、未对齐访问、栈溢出还是函数指针跑飞；再结合最近修改、任务栈高水位、数组越界、空指针、DMA/Cache 一致性等方向排查。RTOS 项目里还要看是不是任务栈太小或 ISR 调用了错误 API。

## 5. 通信接口真题方向

### Q15. UART、IIC、SPI 怎么比较？

**答法：**

- UART：异步串行，常用于日志、调试、模块通信。
- IIC：两线、多设备、速率较低，适合传感器和配置寄存器。
- SPI：同步、高速、时序简单，适合高速外设、Flash、显示、部分数据传输。

工程上不是谁更好，而是看速率、线数、设备数量、距离和硬件连接。

### Q16. IIC 没有 ACK 怎么排查？

**答法：**  
先确认供电、上拉电阻、地址是否正确；再看 SCL/SDA 波形，确认起始停止条件、时钟频率和地址方向位；再确认设备是否处于复位、低功耗或忙状态；最后查驱动时序和 ACK 读取逻辑。

### Q17. SPI 读写数据错位怎么排查？

**答法：**  
先确认 CPOL/CPHA、片选时序、数据位宽和大小端；再看 SCLK、MOSI、MISO、CS 波形；然后确认首字节命令、dummy byte、读写方向和连续传输时序。很多 SPI 问题不是线不通，而是模式或时序边界错。

### Q18. UART 乱码可能是什么原因？

**答法：**  
波特率不匹配、时钟源不准、数据位/停止位/校验位配置不一致、电平标准不匹配、接地问题、缓冲区溢出、接收解析错位都可能导致乱码。

### Q19. CAN 和 UART 有什么区别？

**答法：**  
CAN 是多主总线，带仲裁、错误检测和重发机制，更适合汽车、工业等强可靠通信；UART 更简单，点对点或通过转换模块通信，适合调试和普通串口数据传输。

## 6. DMA / Cache / MPU 真题方向

### Q20. DMA 的价值是什么？

**答法：**  
DMA 可以在外设和内存之间搬运数据，减少 CPU 参与，提高效率。比如 ADC 连续采样、串口大包接收、SPI 数据传输都适合 DMA。CPU 可以在 DMA 搬运期间处理其它任务。

### Q21. DMA 和 Cache 为什么会有一致性问题？

**答法：**  
DMA 直接访问内存，CPU 可能访问 Cache。如果 DMA 写了内存，但 CPU 读到的是旧 Cache，就会出现旧数据；如果 CPU 写了 Cache 但没写回内存，DMA 也可能读不到最新数据。解决方式包括 Cache clean/invalidate、使用 non-cacheable 区域、MPU 配置等。

### Q22. MPU 有什么用？

**答法：**  
MPU 可以配置内存区域的访问权限和属性，比如只读、不可执行、是否 cacheable。它不仅能提升安全性，也能帮助处理 DMA/Cache 一致性问题。

## 7. 调试能力真题方向

### Q23. 程序跑飞怎么排查？

**答法：**  
先看是否进入 HardFault，再看 PC/LR/SP 和 fault 寄存器；检查栈溢出、数组越界、空指针、函数指针、内存踩踏；如果和中断/RTOS 相关，再看中断优先级、ISR API、任务栈和共享资源保护。

### Q24. 系统偶现卡死怎么排查？

**答法：**  
先区分是真死机还是某个任务饿死。可以加心跳、看任务状态、看中断是否还进、看 watchdog 是否复位；RTOS 里重点检查高优先级任务是否一直运行、互斥锁死锁、队列阻塞、临界区过长。

### Q25. 串口协议解析错位怎么查？

**答法：**  
保存原始字节流，先离线找帧头和长度；检查大小端、结构体对齐、长度字段和校验；再看是否丢字节、缓冲区溢出或多线程读写冲突。不要一上来改上层显示。

## 8. 项目深挖真题方向

### Q26. 你这个项目最难的 bug 是什么？

**推荐答法：**  
不要说“没遇到什么 bug”。可以按“现象 - 定位 - 根因 - 解决 - 复盘”讲。比如雷达项目可以讲板端和 Python 侧特征不一致、帧解析错位、目标维护误删；美的实习可以讲显示异常、输入误判、联锁状态异常。

### Q27. 你怎么证明结果是对的？

**推荐答法：**  
不能只说“跑出来了”。跌倒项目看原始采样、FFT、MTI、距离向量、分类概率；多普勒项目看原始帧长度、`rawRadarCube`、FFT 峰值和仿真映射；生命体征项目看点云、目标 ID、微动波形、频谱峰值和目标状态；美的实习看样机功能测试和回归验证。

### Q28. 你做项目时最关注什么工程指标？

**推荐答法：**  
实时性、稳定性、资源占用、可观测性、可维护性。结合项目说：雷达项目关注帧周期内完成处理，模型部署关注 RAM 和推理时间，电控实习关注安全态和异常恢复。

## 9. 大厂面试表达模板

### 模板 1：回答项目贡献

我主要负责的是某条具体链路，而不是整个项目里所有东西。这个链路的输入是什么、输出是什么、中间做了哪些处理、怎么验证正确性，我会按这四步讲。

### 模板 2：回答排查问题

我一般不会直接猜算法错了，而是先分层：硬件/底层是否正常，数据链路是否完整，参数是否一致，最后再看算法和业务逻辑。

### 模板 3：回答工程取舍

这个方案不是绝对最优，而是在当前资源、实时性和开发周期下比较合适。比如 MCU 端不能只看精度，还要看 RAM、Flash、推理时间和调试可观测性。

## 10. 面试前 3 天速背清单

1. C：`volatile`、指针数组、结构体对齐、内存区、函数指针。
2. RTOS：任务状态、调度、队列、信号量、互斥锁、优先级反转、FromISR。
3. MCU：启动流程、中断、HardFault、栈、Flash、DMA。
4. 通信：UART/IIC/SPI/CAN 的区别和典型排查。
5. 项目：每个项目准备一个“链路介绍”、一个“难点”、一个“排查案例”。
