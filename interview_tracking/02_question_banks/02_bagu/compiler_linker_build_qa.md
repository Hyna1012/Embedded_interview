# 编译、链接与工程构建高频问答

> 用途：补充嵌入式面试里经常被追问的编译流程、链接脚本、启动文件、`map` 文件、头文件和工程构建问题。
>
> 这类问题非常适合区分“只会写业务代码”和“理解嵌入式工程”的候选人。

## 1. 使用原则

- 回答编译链接题时，要按“预处理 - 编译 - 汇编 - 链接”这条线讲。
- 回答链接脚本题时，要和 `Flash / RAM / data / bss / stack / heap` 联系起来。
- 回答工程构建题时，要体现你会看报错、看符号、看 `map` 文件，而不是只会点 IDE 下载。

## 2. 高频问答

### Q1. 一个 C 程序从源码到可执行文件大概经历哪些阶段？

**直接回答：**
大致经历预处理、编译、汇编和链接四个阶段。

**展开解释：**
预处理阶段会处理 `#include`、`#define`、条件编译等内容；编译阶段把 C 代码转换成汇编；汇编阶段生成目标文件；链接阶段把多个目标文件、库文件和启动文件合并，解析符号地址，最终生成可执行文件或固件镜像。

**工程补充：**
嵌入式里最后通常还会把 `elf` 转成 `bin` 或 `hex`，用于烧录到 MCU 的 Flash。面试里可以说：“我不只是关注 `.c` 文件，还会关注启动文件、链接脚本和最终镜像布局。”

### Q2. `.h` 文件和 `.c` 文件分别应该放什么？

**直接回答：**
`.h` 文件放对外声明和接口，`.c` 文件放具体实现。

**展开解释：**
头文件通常包含函数声明、类型定义、宏、枚举和模块对外接口；源文件包含函数实现、内部静态变量、私有函数等。头文件被多个源文件包含，所以不适合直接放普通变量定义，否则可能造成重复定义。

**工程补充：**
一个模块比较稳的写法是：头文件暴露最小必要接口，源文件里用 `static` 隐藏内部实现，这样模块边界更清晰。

### Q3. 头文件为什么要加防重复包含？

**直接回答：**
为了避免同一个头文件被多次包含后造成重复定义或编译错误。

**展开解释：**
常见方式是 include guard：

```c
#ifndef MODULE_H
#define MODULE_H

/* declarations */

#endif
```

也可以用 `#pragma once`。它们的目的都是保证头文件内容在一个编译单元里只展开一次。

**工程补充：**
嵌入式工程文件多、依赖复杂，头文件保护是基本习惯。否则一个结构体、枚举或内联函数被重复展开，很容易导致编译问题。

### Q4. 什么是重复定义？什么是未定义引用？

**直接回答：**
重复定义是同一个符号被定义了多次，未定义引用是声明了符号但链接时找不到实现。

**展开解释：**
比如在头文件里直接写 `int g_count = 0;`，多个 `.c` 包含后就会重复定义。正确做法通常是在头文件写 `extern int g_count;`，在某一个 `.c` 文件里真正定义。未定义引用则常见于函数只有声明没有实现、源文件没加入工程、库文件没链接。

**工程补充：**
排查这类问题不要只看报错行，要看链接器报的是哪个符号，再确认符号到底定义在哪里、对应源文件有没有参与构建。

### Q5. `extern` 的作用是什么？

**直接回答：**
`extern` 用来声明一个变量或函数在别的文件中定义。

**展开解释：**
它告诉编译器“这个符号存在，但当前文件不分配存储空间，链接时再去找真正定义”。函数声明默认就有外部链接属性，所以函数前面很多时候不写 `extern` 也可以。

**工程补充：**
全局变量不要滥用 `extern` 到处共享。更好的做法是通过模块接口函数访问内部状态，降低耦合。

### Q6. 静态库和动态库有什么区别？

**直接回答：**
静态库在链接时被打进最终程序，动态库在运行时被加载。

**展开解释：**
静态库常见后缀是 `.a` 或 `.lib`，链接后目标代码成为最终程序的一部分；动态库常见后缀是 `.so`、`.dll`，运行时由系统加载。MCU 裸机项目通常更多接触静态库。

**工程补充：**
嵌入式里厂商 SDK、DSP 库或算法库有时会以静态库形式提供，使用时要注意头文件、库文件、编译器版本和 ABI 是否匹配。

### Q7. 链接脚本是做什么的？

**直接回答：**
链接脚本用来规定代码和数据放到哪些内存区域。

**展开解释：**
它会描述 Flash、RAM 的起始地址和大小，并决定 `.text`、`.rodata`、`.data`、`.bss`、堆、栈等段的布局。比如代码通常放在 Flash，运行时变量放在 RAM，初始化全局变量的初值存在 Flash，启动时复制到 RAM。

**工程补充：**
如果数组太大导致 RAM 溢出，或者 Bootloader 和 App 地址冲突，很多时候就要看链接脚本和 `map` 文件。

### Q8. `.text`、`.data`、`.bss`、`.rodata` 分别是什么？

**直接回答：**
`.text` 放代码，`.rodata` 放只读常量，`.data` 放已初始化全局/静态变量，`.bss` 放未初始化或初始化为 0 的全局/静态变量。

**展开解释：**
`.data` 变量运行时在 RAM，但初始值要存放在 Flash，启动时复制过去；`.bss` 不需要在 Flash 存每个 0，启动时统一清零即可。

**工程补充：**
理解这些段，有助于解释为什么大数组会占 RAM，为什么启动文件要复制 `.data` 和清零 `.bss`。

### Q9. `map` 文件有什么用？

**直接回答：**
`map` 文件可以查看最终固件的内存布局、符号地址和各模块占用大小。

**展开解释：**
它能告诉你每个函数、变量、目标文件被放到了哪里，占了多少 Flash/RAM。遇到内存超限、符号冲突、函数地址异常时，`map` 文件非常有用。

**工程补充：**
面试里可以说：“如果项目 Flash 或 RAM 超限，我会先看 `map` 文件，找出大数组、库函数或异常增长的模块，再决定优化方向。”

### Q10. 编译优化等级可能带来什么影响？

**直接回答：**
优化等级会影响代码大小、运行速度和调试可见性。

**展开解释：**
高优化可能内联函数、删除未使用变量、重排指令，让单步调试看起来和源码不完全一致；低优化更利于调试，但代码可能更慢或更大。

**工程补充：**
如果 Debug 模式正常、Release 模式异常，要重点怀疑未初始化变量、越界、`volatile` 缺失、时序依赖和优化暴露出的未定义行为。

### Q11. 为什么有时加了 `volatile` 之后问题就没了？

**直接回答：**
因为原来变量可能被编译器优化缓存，导致程序读不到外部变化。

**展开解释：**
中断、硬件寄存器、DMA 状态、线程共享标志都可能在当前执行流之外被修改。如果不加 `volatile`，编译器可能认为变量不会变，从而把读取优化掉。

**工程补充：**
但要强调：`volatile` 只解决“可见性”，不解决原子性和互斥。如果是多任务竞争，还需要临界区或锁。

### Q12. Makefile 或构建系统主要解决什么问题？

**直接回答：**
构建系统负责描述哪些源文件参与编译、使用哪些编译选项、如何链接生成目标文件。

**展开解释：**
它会管理依赖关系，避免每次全量编译，也能统一编译器、宏定义、头文件路径、库路径、链接脚本等配置。

**工程补充：**
即使用 IDE，也要理解背后其实是在调用编译器、汇编器、链接器和烧录工具。这样遇到构建错误时才不会完全依赖图形界面。
