# 嵌入式面试细节详细回答

> 这份文件只保留项目、实习、行为面和工程闭环相关的“细节题”。
> 回答结构默认统一为：
> 1. 先直接回答
> 2. 再展开解释
> 3. 最后补工程补充

## 1. 跌倒项目

### Q1. 你先完整介绍一下跌倒检测项目。

**直接回答：**
这个项目是一个基于 `24GHz` 毫米波雷达的端侧跌倒检测系统，目标是在设备本地完成从采集、处理到识别输出的完整闭环。

**展开解释：**
硬件平台是 `RKB2301`，主控 MCU 是 `AT32F435`，雷达前端芯片是 `SRK2401`。整个系统不是只做数据采集，也不是只做算法离线验证，而是把板端采样、信号处理、特征提取、轻量化模型推理和上位机联调全部串起来。

我在这个项目里负责的内容比较完整，前端包括板级初始化、ADC 数据采集链路、AFE 配置和底层驱动调试；中间包括基于 `CMSIS-DSP` 的信号处理流程，比如均值去除、窗函数、`FFT`、静态杂波抑制和距离特征提取；后端包括 `CNN / GRU / LSTM` 模型训练、参数导出、端侧部署以及 `TLV` 协议设计和上位机联调。

**工程补充：**
这个项目最能体现的不是单一算法能力，而是“板端采集 - 信号处理 - 模型部署 - 联调验证”的完整闭环能力。

### Q2. 你在跌倒项目中的最大困难是什么？

**直接回答：**
我觉得最大困难是，怎么在 MCU 端把信号处理链路和模型推理真正对齐起来。

**展开解释：**
这个项目难的地方不只是训练一个模型，而是前面有采样、`FFT`、静态杂波抑制、特征提取，后面又有滑窗、时序输入、模型推理和结果输出。只要板端输入和训练时输入在特征顺序、归一化方式、窗口长度或者时序组织上有一点不一致，最后识别效果就会明显下降。

我当时不是只看最后分类结果，而是把问题拆成多层去定位。先对比板端和 Python 侧的中间特征，再看滑窗之后送进模型的输入张量是不是一致，最后才看最终识别结果。这样能更快判断问题是出在预处理、特征构造还是模型部署。

**工程补充：**
这道题后面很容易被追问，所以你要记住一句核心话术：
“最大困难不是把模型跑起来，而是让板端输入和训练输入真正对齐，并保证端侧结果可信。”

### Q3. 为什么要设计 `TLV` 协议？

**直接回答：**
因为项目里输出的数据类型不止一种，`TLV` 协议更适合调试和扩展。

**展开解释：**
项目里除了原始采样数据，还有中间处理结果、距离特征、分类概率和状态信息。如果按固定裸数据流发，上位机解析容易混乱，后续加字段也很麻烦。`TLV` 的好处是每段数据都有类型和长度，上位机可以根据类型做分发和解包。

**工程补充：**
它的价值不在“高级”，而在于联调时可观测、可扩展、好定位问题。

### Q4. 为什么模型最终更偏向 `LSTM`？

**直接回答：**
因为跌倒识别本质上是一个和时序过程强相关的任务，`LSTM` 对动作过程建模更合适。

**展开解释：**
我在项目里对 `CNN`、`GRU` 和 `LSTM` 都做过对比。`CNN` 部署方便，但对纯时间关系表达通常不如循环结构；`GRU` 和 `LSTM` 都更适合建模时间变化，其中 `LSTM` 在当前任务里对动作过程的表达更充分。

最终选择时，我不是只看精度，而是一起看识别效果、时序建模能力、推理时延和 RAM 占用。项目当前版本更偏向 `LSTM`，主要是因为它在当前数据和任务上更平衡。

**工程补充：**
一定要强调这是工程权衡，而不是“谁最强就上谁”。

## 2. 多普勒项目

### Q5. 多普勒项目和跌倒项目最大的区别是什么？

**直接回答：**
跌倒项目更偏端侧识别和模型部署，多普勒项目更偏雷达底层驱动和原始数据采集链路。

**展开解释：**
多普勒项目的目标不是做最终分类，而是围绕 `24GHz FMCW` 雷达完成底层采集链路和距离-速度验证。它的核心是把 `AFE / ADC` 参数配置、`frame/chirp` 采样时序、ADC 采样完成事件、`rawRadarCube` 数据组织和原始帧输出串起来，再结合 Python 侧解包和二维 FFT 验证距离-速度映射是否一致。

**工程补充：**
这也是你简历里两个项目互补的关键：跌倒讲端侧识别，多普勒讲底层驱动和采集数据源。

### Q6. 为什么要做 Python 仿真？

**直接回答：**
因为仿真能帮助验证板端结果是不是和理论关系一致。

**展开解释：**
我在 Python 里建立了目标距离、目标速度、`IF` 频率和多普勒频率之间的映射关系，再把仿真结果和采集数据处理结果做对照。这样如果热图峰值位置不对，我就能更快判断是采样参数、原始帧解析还是距离速度换算出了问题。

**工程补充：**
这个点很适合体现“我不只是写代码，我还会建立验证闭环”。

## 3. 美的实习

### Q7. 你在美的实习主要做什么？

**直接回答：**
我在美的实习做的是微波炉电控方向，核心是围绕主控板做整机电控逻辑开发和联调。

**展开解释：**
这段实习不是单纯写一个底层驱动，而是围绕主控板的软件逻辑、状态机、安全联锁、继电器控制和整机联调来做。也就是说，我的工作重点是让主控板对输入、状态和输出的组织更稳定、更符合整机要求。

**工程补充：**
面试时一定要强调“整机电控逻辑”，不要把这段经历讲成调几个外设。

### Q8. 什么是整机联调？你当时是怎么做的？

**直接回答：**
整机联调就是把控制板、功率板、显示、输入和执行部件一起带起来，在真实场景下验证状态机和控制逻辑是否成立。

**展开解释：**
我做整机联调时，不是只看代码逻辑，而是把控制板、功率板、门开关、显示和执行部件一起放到样机环境里跑。先确认单板基础功能正常，再按待机、启动、运行、暂停、开门、故障这些场景逐项验证，看输入、状态切换、显示和输出动作是不是一致。

**工程补充：**
最加分的一句是：
“整机问题很多时候不是单板问题，而是系统联动问题。”

### Q9. 为什么你会把“问题闭环”当成这段实习的核心收获？

**直接回答：**
因为产品研发里更核心的不是一次把功能写出来，而是把问题发现、定位、修掉并稳定下来。

**展开解释：**
真实产品开发里，很多时间并不是在从零写新功能，而是在样机联调、中试、小批阶段不断处理各种异常。比如输入识别不稳定、状态切换不一致、整机动作不对、保护逻辑边界不清。这些问题如果不闭环，系统就无法真正进入可量产、可维护的状态。

**工程补充：**
这个回答很适合用来体现你和纯校园项目同学的差异。
